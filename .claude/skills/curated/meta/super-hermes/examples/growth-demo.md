# Growth Loop Demo: Before vs After

*This demonstrates the growth mechanism. Same code, same skill, different results because the agent learned from its previous blind spots.*

---

## Run 1: Fresh analysis (no .prism-history.md)

The agent checks for history, finds none, generates a lens focused on what it observes:

> **Generated Lens:** "Map every state transition with its synchronization: which lock is held, what writes occur. The state property is impure — it mutates _state and _half_open_calls during a read without holding the lock..."

**Result:** Deep analysis of thread safety and race conditions. 1,047 words. Conservation law: `Fault Isolation × State Correctness = Constant`.

**Constraint report says:** Maximized thread-safety. Sacrificed performance, observability, testing, API design.

## Run 2: After /prism-reflect saved constraint history

The agent reads .prism-history.md and adjusts:

> **Step 0:** "Found .prism-history.md. Previous analysis maximized thread-safety and race conditions. Sacrificed: performance under load, observability, testing ergonomics, API design quality."
>
> **"→ This lens will target those sacrificed dimensions."**
>
> **Generated Lens:** "Performance + Observability + API Quality Lens — For each public method, construct a high-load scenario (10,000 concurrent calls) and trace the bottleneck. Identify every silent failure path. Build test cases that exercise each state transition..."

**Result:** Completely different analysis covering performance ceilings, observability gaps, and untestable behaviors. 1,257 words. Conservation law: `Encapsulation × Observability = Constant` — the circuit breaker hides state behind properties and private fields for API simplicity, making it unobservable and untestable; every fix (add `get_stats()`, expose `retry_after`, inject time provider) trades encapsulation for observability. Different findings.

| Dimension | Scan 1 (no history) | Scan 2 (with history) |
|-----------|--------------------|-----------------------|
| Lens focus | Thread safety, race conditions | Performance, observability, API quality |
| Words | 1,047 | 1,257 |

| Step 0 | "No history found" | "Found history → targeting sacrificed dimensions" |

**The agent grew.** Same code. Same skill. Different analysis. The second run covered what the first run missed — because the constraint history told it where its blind spots were.
